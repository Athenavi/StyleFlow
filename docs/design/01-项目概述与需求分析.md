# 01 - 项目概述与需求分析

> **版本**：v1.0 | **更新**：2026-06-26

---

## 1.1 项目背景

服装行业数字化过程中面临四大痛点：

| 痛点 | 现状 | 期望 |
|------|------|------|
| **设计效率低** | 设计师手绘/PS 出款，改版周期长 | AI 辅助文生图、款式延展，秒级出稿 |
| **ERP 数据孤岛** | 设计、打版、生产、财务系统割裂，反复录入 | 设计与 ERP 数据打通，自动同步 |
| **真人模特成本高** | 拍摄样衣需请模特、摄影棚、后期修图 | AI 虚拟试衣，一键合成穿搭效果 |
| **核工价繁琐** | 工艺员手工核对工序、工时，易出错 | 自动匹配工序模板，智能核算工价 |

**StyleFlow** 定位为"AI 服装设计-生产协同平台"，从企划、设计、虚拟试衣到核工价、计件工资，打通全链路自动化。

---

## 1.2 目标用户

| 角色 | 职责 | 核心诉求 |
|------|------|----------|
| **设计师** | 款式设计、改版 | 快速出图、AI 灵感激发、系列延展 |
| **版师 (Pattern Maker)** | 打版、工艺单制作 | 自动生成工艺初稿、减少手动录入 |
| **工艺员** | 核算工价、编制作业指导书 | 自动匹配工序模板、减少重复计算 |
| **生产管理人员** | 查看流程进度、审批 | 可视化流程看板、待办推送 |
| **财务/HR** | 计件工资核算 | 自动汇总报工数据、生成工资表 |
| **系统管理员** | 用户管理、配置维护 | 权限管理、日志审计、系统配置 |

## 1.3 核心功能列表

### P0 — MVP（必须实现）

| 模块 | 功能 | 说明 |
|------|------|------|
| **用户系统** | 注册/登录/角色管理 | JWT 认证，Django Admin 管理用户 |
| **AI 设计工坊** | 文生图 | 输入 Prompt → 调用 SD API → 返回图片 |
| **AI 设计工坊** | 图生图（款式延展） | 上传参考图 → ControlNet 处理 |
| **AI 设计工坊** | 设计稿管理 | 图片列表、收藏、标签、搜索 |
| **虚拟试衣** | 单人换装合成 | 人物图 + 服装图 → VTON 模型输出 |
| **任务异步处理** | Celery 任务队列 | AI 生成异步执行，前端轮询/SSE 获取结果 |

### P1 — 核心业务（第二阶段）

| 模块 | 功能 | 说明 |
|------|------|------|
| **ERP 对接** | 数据同步（款号、BOM、工序） | Celery Beat 定时同步 |
| **商品企划** | AI 辅助生成企划案 | LLM 分析波段/品类数据，生成灵感板 |
| **工艺单生成** | 自动提取设计属性，生成工艺单 | LLM Function Calling 结构化输出 |
| **工作流引擎** | 企划→设计→工艺→核价→审批 | 基于状态机的流程驱动 |
| **核工价** | 自动匹配工序模板，核算标准工价 | 工序库 + 工价规则引擎 |

### P2 — 增强功能（第三阶段）

| 模块 | 功能 | 说明 |
|------|------|------|
| **计件工资** | 生产报工 → 工资计算 | 对接 ERP 报工数据 |
| **微信小程序** | 移动端核心操作（审批、拍照上传） | Taro 跨端开发 |
| **数据分析看板** | 设计产出统计、成本分析 | ECharts / Ant Design Charts |
| **多模型支持** | 可切换不同 LLM / 图像模型 | 统一 AIService 抽象层 |

## 1.4 非功能需求

| 类别 | 要求 | 说明 |
|------|------|------|
| **性能** | 图片生成 < 30s（SD 端到端） | 异步队列 + 前端进度提示 |
| **性能** | API 响应 < 500ms（非 AI 请求） | 数据库索引 + Redis 缓存 |
| **可用性** | 核心功能可用时间 > 99.5% | Docker 容器化，健康检查自动重启 |
| **安全性** | JWT 鉴权，HTTPS，敏感字段加密 | 遵循 OWASP Top 10 |
| **扩展性** | 新增 AI 模型 < 2 人天 | 工厂模式接入，配置化切换 |
| **可维护性** | 代码测试覆盖率 > 70% | pytest + Django TestCase |
| **兼容性** | 支持 Chrome/Firefox/Edge 最新版 | 响应式布局，移动端适配 |

## 1.5 约束与假设

- **单人开发**：架构追求极简，避免过度设计（参照 Ponytail 原则）
- **部署环境**：初期单机 Docker Compose，后期可拆分
- **ERP 对接**：假设目标 ERP 提供数据库只读账号或 API，具体对接方案在 [08-ERP对接方案](./08-ERP对接方案.md) 细化
- **AI 模型**：假设有 GPU 服务器（至少 16GB VRAM）运行 SD 和 VTON 模型；若无则切换到云端 API 方案
- **数据库**：PostgreSQL 15（不支持 SQLite 作为生产库）
